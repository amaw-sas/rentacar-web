# AMEND 2026-07-04 — issue #221: live city count → deterministic source of truth

Companion amendment to `scenarios/city-count-derivation.scenarios.md` (which is
write-once under the SDD guard, so the change is recorded here instead of edited
in place). This note **supersedes the "live" wording** of SCEN-001, SCEN-003 and
SCEN-005 of that spec.

## What changed and why

Operator correction #3 derived the "N ciudades" figure — and the footer/home city
`v-for` lists — from **live** `useFetchRentacarData().cities`, so the number
self-corrected from the dashboard. Issue #221 found that this SAME live derivation
caused **intermittent hydration mismatches** on ISR pages:

- The 19 city routes and `/` are `isr: 3600`. Nuxt hydrates an ISR route from a
  separately-cached `_payload.json`. The cached HTML and that payload can capture
  **different `cities` snapshots** (independent revalidation windows, or a city
  added/removed between the two renders).
- The footer/home then rendered a different number of `<a>` nodes / a different
  count on the server than the client reconstructed → `[Vue warn] Hydration …
  mismatch` (never in dev, where SSR is fresh per request — matching the #221
  reproduction notes).

The two requirements are **mutually exclusive** for an SSR text node under ISR: a
number cannot both track a value that may drift and be drift-proof.

## Resolution (directive-approved)

Keep operator #3's **durable invariant** — ONE guarded source of truth, never
scattered hardcoded literals — but move that source from live runtime data to a
**deterministic build-time constant**:

- `SERVICE_CITIES` (`packages/logic/src/utils/serviceCities.ts`): the 19 active
  service cities (`{id: slug, name}`), the same set enumerated as `isr` routes in
  every brand's `routeRules`.
- `useCityCount()` now returns `SERVICE_CITIES.length` (drift-proof; still a
  `ComputedRef<number>`, so every `{{ cityCount }}` call site is unchanged).
- The footer (`layouts/default.vue` ×3) and home `Cities.vue` (×2) `v-for` over
  `SERVICE_CITIES`.

Server render and client's first render are now byte-identical on these subtrees
→ no mismatch, and the links stay crawlable in SSR (a `<ClientOnly>` fallback
would have dropped them and lost the SEO).

## Invariant preserved / re-read

- SCEN-002 / SCEN-004 (no hardcoded literal in Vue-rendered copy; every mention
  derives from `useCityCount`) — **unchanged and still enforced** by
  `packages/ui-alquilame/tests/city-count.test.ts`. Components still bind
  `{{ cityCount }}`; no `\d+ ciudades` literal was introduced.
- SCEN-001 / SCEN-003 — wherever they say the figure equals the **live**
  `cities.length`, read "equals `SERVICE_CITIES.length` (the canonical count)".
  The observable Given/When/Then (SSR-present, CLS-safe, positive plural count) is
  unchanged; only the source moved from runtime to a guarded build-time constant.
- SCEN-005 (footer `v-for` still renders, SSR-present, 0 console errors) —
  **strengthened**: the list is now drift-proof, closing the #221 mismatch.

## Cost / operational note

Adding or removing a city is now a **coordinated edit of three lockstep places**:

1. `SERVICE_CITIES`,
2. the `isr: 3600` routes in each brand's `nuxt.config.ts` `routeRules`,
3. the build-time SEO count literals (`nuxt.config.ts` `description`,
   `app.config.ts`, `pages/gana/index.vue`).

Adding a city already required (2), so this does not add a new class of manual
work. Guarded by `serviceCities.test.ts` and `useCityCount.test.ts`.
